# 📘 **Semantic Cube Runtime
構築指示書２（注釈＋プロンプトシード集）**

---

# 1. 🌱 **関数型志向で実装する際の重要な注釈**

v1.2 で「関数型志向・疎結合・ステートレス」を採用したけど、  
実装時に特に気をつけるべきポイントを整理する。

---

## ◆ 1-1. **“状態を持つのは KVHub だけ” を徹底する**

- CubeFactory → pure  
- TrustEvaluator → pure  
- PartialEngine → pure  
- IntegrationEngine → pure  
- ConflictEngine → pure  
- DefragEngine → pure  

**唯一の例外：KVHub（世界の記憶）**

→ これが守られていれば、  
- 並列実行  
- 再現性  
- テスト容易性  
- 依存関係の明確化  

がすべて成立する。

---

## ◆ 1-2. **pure function の条件**

- 入力が同じなら出力も同じ  
- 外部状態に依存しない  
- 外部状態を書き換えない  
- 副作用を持たない  

特に **LLM 呼び出しは pure function として扱う**  
（外部APIだが、内部状態を持たないため pure とみなす）

---

## ◆ 1-3. **KVHub は“世界の状態”であり、“論理の状態”ではない**

KVHub が保持するのは：

- キューブの集合  
- その trust / orientation / grid  
- 履歴（turn, ref_count）  
- cluster の構造

保持しないもの：

- 「統合すべきか」  
- 「葛藤すべきか」  
- 「採用すべきか」  
- 「判断軸」  

→ これらは **LLM の仕事**。

---

# 2. 🔥 **閾値レス設計の重要な注釈**

v1.2 で「閾値を持たない」方針を採用した理由を、  
実装者向けに明確化しておく。

---

## ◆ 2-1. **プログラム側は“物理法則”だけを担当する**

- grid距離  
- orientation距離  
- layer距離  
- semantic_trust（距離の指数減衰）  

これらは数学的に決まるため、  
**プログラム側で閾値を持っても問題ない領域**。

---

## ◆ 2-2. **しかし“判断”は LLM に委譲する**

- 高トラストか  
- 低トラストか  
- 統合すべきか  
- 葛藤すべきか  
- 採用すべきか  
- 保留すべきか  

これらは **意味判断・価値判断**であり、  
プログラム側が閾値を持つと **人間の判断を固定化してしまう**。

→ だから **trust 分布を LLM に渡し、判断は LLM に任せる**。

---

## ◆ 2-3. **Test4（低トラスト葛藤）は閾値レス設計の真価が出る**

- trust が低い  
- 方向はバラバラ  
- しかし「採用すべきでない」という結論は LLM が出す  

→ プログラム側は「低トラスト判定」を持たない  
→ LLM が trust 分布を見て判断する  
→ これが **Semantic Cube Architecture の“意味判断の分離”** の核心

# 3. 🧩 **役割プロンプト（シード案）**

v1.2 で提示したものを、  
**実際に使いやすい形に整形した“本番用シード”**としてまとめる。

---

## ◆ 3-1. PartialEngine（部分回答生成）

```
あなたは参謀です。
以下の【1つの根拠】だけを絶対的な前提として、
ユーザーの質問に部分的に答えてください。

- 他の視点には触れない
- 新しい主張を追加しない
- この根拠の視点だけで答える
```

---

## ◆ 3-2. IntegrationEngine（統合ルート）

```
以下の部分回答は互いに矛盾がなく、同じ方向性を補完しています。
これらを自然に統合し、一貫した最終提案を作成してください。

- 新しい主張は追加しない
- 部分回答の内容だけで統合する
- 方向性の一貫性を保つ
```

---

## ◆ 3-3. ConflictEngine（高トラスト葛藤）

```
以下の部分回答には、方向性の異なる主張があります。
無理に統合すると矛盾を生むため、以下の方針で回答してください。

- 対立点と共通点を整理する
- それぞれの視点が成立する条件を説明する
- 統合は行わず、選択肢として提示する
- 最後に判断軸を示す
```

---

## ◆ 3-4. ConflictEngine（低トラスト葛藤：Test4）

```
以下の部分回答は互いに対立していますが、
いずれも信頼性（トラストスコア）が低く、
結論の根拠として採用すべきではありません。

- なぜ採用できないのか（信頼性の低さ）を説明する
- 追加の情報やより信頼できる根拠が必要であることを示す
- 結論は保留とし、判断を急がないよう助言する
```

---

## ◆ 3-5. DefragEngine（デフラグ）

```
以下のキューブ群は、類似した方向性を持つクラスタです。
このクラスタの中心的な意味を抽象化し、
新しい「Origin」としてふさわしい1文を生成してください。

- クラスタ全体の共通点を抽象化する
- 具体的すぎる要素は排除する
- 価値観・方向性の核を1文にまとめる
```

# 4. 🌙 **第1弾で伝えきれなかった重要ニュアンス（v2.0補足）**

---

## ◆ 4-1. **Semantic Cube Runtime は“意味の物理エンジン”である**

- grid  
- orientation  
- trust  
- cluster  
- defrag  

これらは **意味空間の物理法則**であり、  
“判断”ではなく“現象”を扱う。

---

## ◆ 4-2. **LLM は“意味判断エンジン”である**

- 統合  
- 葛藤  
- 採用  
- 保留  
- 抽象化  

これらは **人間の判断に近い領域**であり、  
プログラム側が決めるべきではない。

---

## ◆ 4-3. **Test1〜Test5 は“世界線の連続現象”**

- Test1：初期化  
- Test2：群れのダイナミクス  
- Test3：高トラスト葛藤  
- Test4：低トラスト葛藤  
- Test5：デフラグ  

これらは **独立したテストではなく、  
1つの意味空間の中で連続して起こる現象**。

---

## ◆ 4-4. **デフラグは“意味空間の進化”**

- cluster の重心  
- 抽象化  
- 新Origin生成  

これは単なるクラスタリングではなく、  
**意味空間の自己再編成（self-organizing system）**。

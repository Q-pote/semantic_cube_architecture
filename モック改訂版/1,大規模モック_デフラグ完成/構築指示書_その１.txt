# 📘 **Semantic Cube Runtime v1.2  
構築仕様書（関数型志向＋閾値レス対応版）**

君の追加要件を反映したポイントは以下の3つ：

---

# 🌱 **追加要件（反映済み）**

## ◆ 初期化について（関数型志向）
- **各モジュールはステートレス関数として実装する**  
- オブジェクト指向の内部状態（mutable state）は禁止  
- KVHub だけは「世界の状態」を保持するため例外的に stateful  
- それ以外は **すべて純粋関数（pure function）** として設計

→ これにより **並列呼び出し・再現性・テスト容易性** が向上する。

---

## ◆ 全体の挙動について（閾値レス）
- **LLM の判断を最優先**  
- プログラム側の閾値（threshold）は極力持たない  
  - 高トラスト／低トラストの判定  
  - 葛藤の採用／不採用  
  - 意見の重み付け  
- ただし、  
  - grid距離  
  - 角度距離  
  - layer距離  
 などの「純粋なベクトル空間計算」は例外（数学的に必要）

→ つまり、**最終判断は LLM に委譲する**  
（プログラムは“物理法則”だけを担当する）

---

## ◆ モジュール間は疎結合・ステートレス
- main → 各モジュールは **関数呼び出しのみ**  
- モジュール間で状態を共有しない  
- KVHub だけが「世界の記憶」として存在  
- 他はすべて「入力 → 出力」の純粋関数

5. テストシナリオ（5本のテストの狙い）
Test1：代表選抜（初期キャリブレーション）
目的：
grid 正規化が正常か
trust が自然なレンジで出るか
「最も意味が近い1つ」が正しく選ばれるか

期待：
Top1 が意味的に妥当
trust が 0.2〜0.4 付近で分布
JSONダンプで内部空間に群れが散らばっている


Test2：順次回答 → 統合／葛藤分岐（高トラスト前提）
目的：
複数の高トラスト根拠から部分回答を生成し、
方位差が小さい → 統合ルート
方位差が大きい → 葛藤ルート
という分岐が正しく動くかを見る。

期待：
Partial #1〜#k がそれぞれ異なる視点を持つ
葛藤なし → 統合プロンプトで一貫した提案
葛藤あり → 対立整理＋選択肢提示


Test3：高トラスト葛藤（カレー戦争など）
目的：
話題は同じ・trust は高い・方向だけ真逆  
というケースで、葛藤検知が確実に発火し、
統合を禁止して「対立の可視化＋判断軸提示」ルートに入れるかを見る。

期待：
Top-k がすべて高トラスト
最大方位差 ≒ 1.0
統合ルートには絶対行かない
最終回答は「選択肢」と「判断軸」を提示


Test4：低トラスト葛藤（採用すべきでない根拠群）
ここが今回の一番大事な修正ポイント。

見たいもの：
「葛藤する根拠が複数あるが、どれもトラストスコアが低い」
その結果として、
「これらの意見は採用すべきではない」
というメタ判断をLLMが出せるかどうか。

条件設計のイメージ：
Query に対して意味的に遠い／ノイズっぽいキューブ群を意図的に混ぜる
それらの間では方向がバラバラ or 軽く対立している
しかし 全体として trust が低いレンジ（例：0.05〜0.15）に収まる

期待する挙動：
方位差的には「葛藤あり」に見えるが、
trust 分布を見たときに
「高トラスト群ではない」
「どの根拠も採用基準を満たさない」
という判定ができる。

最終回答は：
「複数の対立する意見は存在するが、いずれも信頼性が低く、結論の根拠として採用すべきではない」
「追加の情報やより信頼できるデータに基づいて判断すべき」
といった “採用保留／却下” のメタ判断 になる。


Test5：デフラグ（再編成）
目的：
これまでの Query / 群れ / trust / cluster の履歴をもとに、
似た方向のキューブ群をクラスタリングし、
その重心から新しい Origin を生成できるかを見る。

期待：
DefragEngine が cluster を受け取り、
新しい「抽象度の高い Origin 文」を生成
それを KVHub に put し、
以降の Query で「新Origin」が強い重力源として振る舞う。
